# Notification Service

Небольшой сервис уведомлений на Laravel 12.

Что умеет:

- принимает пачку уведомлений через API;
- создает отдельное уведомление для каждого получателя;
- не создает повторную пачку при одинаковом `idempotency_key`;
- кладет уведомления в RabbitMQ;
- обрабатывает очередь отдельным воркером;
- учитывает приоритет `high/default/low` через priority queue RabbitMQ;
- делает retry при ошибке провайдера и переводит уведомление в `dropped` после лимита попыток.

`idempotency_key` должен приходить от клиента, который вызывает API.  
Он нужен, чтобы повторный запрос по тому же бизнес-событию не создал вторую пачку уведомлений.

Например, если клиент не получил ответ из-за timeout, он может повторить тот же запрос с тем же `idempotency_key`, и сервис вернет уже созданную пачку.

Внешний SMS/email-провайдер здесь заменен на fake-реализацию. Она нужна, чтобы проверить основной процесс, retry и статусы без подключения настоящего сервиса.

## Запуск

```
docker compose up --build
```

После запуска:

- `http://localhost:8080/` - краткая информация о сервисе;
- `http://localhost:8080/api` - список основных endpoint'ов;
- `http://localhost:8080/up` - health-check Laravel;
- `http://localhost:15673` - RabbitMQ UI (`guest` / `guest`).

Воркер очереди запускается отдельно:

```
docker compose exec app php artisan notifications:consume
```

Он висит в консоли и ждет сообщения из RabbitMQ. Остановить можно через `Ctrl+C`.

## API

Создать пачку уведомлений:

```
curl -X POST http://localhost:8080/api/notifications/bulk \
  -H "Accept: application/json" \
  -H "Content-Type: application/json" \
  -H "X-Api-Key: local-demo-key" \
  -d '{
    "idempotency_key": "route-change-1",
    "channel": "sms",
    "priority": "high",
    "message": "Изменился маршрут",
    "recipient_ids": ["driver-1", "driver-2"]
  }'
```

`idempotency_key` задает клиент API. Для одного и того же события нужно отправлять один и тот же ключ.

Для локального запуска есть демо-ключ:

```
X-Api-Key: local-demo-key
```

Новый ключ в этом варианте добавляется через `.env`:

```
NOTIFICATION_DEMO_API_KEY=another-local-key
```

В реальном сервисе такие ключи лучше хранить в БД или отдельном secrets-хранилище, но для тестового задания этого достаточно.

Пример:

- первый запрос: `idempotency_key = route-change-1`;
- новое событие: уже другой ключ, например `route-change-2`.

Если повторить POST-запрос с уже использованным `idempotency_key`, новая пачка не создается. В ответе вернется тот же `batch_id` уже созданной пачки.
В этой реализации ключ считается общим для всего сервиса, поэтому разные клиенты не должны использовать одинаковые ключи для разных событий.

Посмотреть историю получателя:

```
curl http://localhost:8080/api/subscribers/driver-1/notifications \
  -H "X-Api-Key: local-demo-key"
```

После обработки воркером статус должен поменяться с `queued` на `delivered`.

Для проверки retry можно отправить сообщение с текстом, где есть `fail`. Fake-провайдер будет падать, а уведомление после нескольких попыток перейдет в `dropped`.

## Тесты

```
docker compose exec app php artisan test
```

В тестовом задании отдельно запрошены интеграционные тесты, поэтому здесь проверяются основные сценарии целиком: API, БД, сервисный слой, доставка и порядок обработки приоритетов RabbitMQ. Unit-тесты для простых правил бизнес-логики отдельно не добавлялись, чтобы не раздувать объем задания.

## Что внутри

- Laravel 12, PHP 8.4;
- PostgreSQL для хранения пачек и уведомлений;
- RabbitMQ для очереди отправки;
- Redis используется для временной защиты от параллельных дублей по `idempotency_key` и простого rate limit на создание пачек;
- рабочие API-методы требуют заголовок `X-Api-Key`, демо-ключ для локалки: `local-demo-key`;
- nginx + php-fpm работают в одном app-контейнере через supervisord;
- тесты используют SQLite in-memory, чтобы не трогать локальную PostgreSQL.

Основные статусы уведомления:

- `queued` - создано и ждет обработки;
- `sent` - передано провайдеру;
- `delivered` - доставлено;
- `dropped` - автоматические попытки закончились, дальше нужен ручной разбор.
